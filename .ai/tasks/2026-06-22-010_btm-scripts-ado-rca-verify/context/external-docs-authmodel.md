## Q1 — `$(System.AccessToken)` identity: Project "Build Service", independent of agent pool

**VERDICT: CONFIRMED (A1).** `System.AccessToken` carries the per-job security token; its
permissions derive from the **Build Service** identity (project-scoped `{Project} Build Service ({Org})`
or collection-scoped `Project Collection Build Service ({Org})`), NOT from the agent pool.
The token/identity is generated and resolved server-side by Azure Pipelines, so it is the same
whether the job runs on a Microsoft-hosted or self-hosted agent. The agent merely *receives and
uses* the token; it does not mint or alter its identity.

- URL: https://learn.microsoft.com/azure/devops/pipelines/process/access-tokens?view=azure-devops
  - Quote: "A **job access token** is a security token that is dynamically generated by Azure
    Pipelines for each job at run time. The agent on which the job is running uses the job access
    token in order to access these resources in Azure DevOps."
  - Quote: "The token's permissions are derived from (a) job authorization scope and (b) the
    permissions you set on project or collection build service account."
  - Quote (Scoped build identities): "The **project-scoped identity** name has the following
    format: `{Project Name} Build Service ({Org Name})` … For example, if the organization name
    is `fabrikam-tailspin` and the project name is `SpaceGameWeb`, this account has the name
    `SpaceGameWeb Build Service (fabrikam-tailspin)`."
- URL: https://learn.microsoft.com/azure/devops/pipelines/build/variables?view=azure-devops#systemaccesstoken
  - Quote: "`System.AccessToken` is a special variable that carries the security token used by
    the running build."
- URL: https://learn.microsoft.com/azure/devops/release-notes/roadmap/2025/new-service-connection
  - Quote: "Tasks in a pipelines use either a job access token, available via the built-in
    variable `System.AccessToken` … the permissions of this token are based on the Project Build
    Service identity, meaning all job access tokens in a project have identical permissions."

**Independence from agent pool:** A1 supports this *indirectly but unambiguously* — the token is
"dynamically generated by Azure Pipelines for each job at run time," and its permissions are a
function of (job authorization scope) + (build service account permissions). Neither input is
the agent pool. Official docs do not contain a single sentence literally asserting "the identity
is independent of Microsoft-hosted vs self-hosted"; that specific phrasing is **A2 INFER** from
the mechanism above (token minted server-side, agent only consumes it). Confidence: high.
The confirm/deny target — "it is the project Build Service identity AND independent of the agent
pool" — is therefore: project Build Service identity = **CONFIRMED**; agent-pool independence =
**INFER (strong)**.

## Q2 — `enforceJobAuthScope=true` ("Limit job authorization scope to current project"): what is lost; can it affect same-project Boards work items?

**VERDICT: CONFIRMED (A1).** Setting the scope to **project** restricts the token to resources in
the *same project as the pipeline* — it removes access to **all other projects'** resources
(repos, feeds, etc.). It is **not limited to repos**: the project-scoped Build Service identity's
permissions govern what it can do, and that explicitly includes work-item operations. Docs call
out restricting the pipeline "from creating work items" as a managed scenario, and the
project-scoped identity "may not have permissions to a resource that the collection-scoped one
did have." Within the SAME project, the token's ability to read/create/update work items is
governed by the **`{Project} Build Service ({Org})`** account's permissions (project-level and
area-path level), not by the scope toggle per se — the scope toggle changes *which* identity
(project vs collection) is used, and the project identity may legitimately lack a permission the
collection identity had.

- URL: https://learn.microsoft.com/azure/devops/pipelines/process/access-tokens?view=azure-devops#job-authorization-scope
  - Quote: "You can set the job authorization scope to be **collection** or **project**. By
    setting the scope to **collection**, you choose to let pipelines access all repositories in
    the collection or organization. By setting the scope to **project**, you choose to restrict
    access to only those repositories that are in the same project as the pipeline."
- URL: https://learn.microsoft.com/azure/devops/pipelines/process/access-tokens?view=azure-devops#manage-build-service-account-permissions
  - Quote: "One result of setting project-scoped access may be that the project-scoped identity
    may not have permissions to a resource that the collection-scoped one did have."
  - Quote: "You may want to change the permissions of job access token in scenarios such as the
    following: 1. You want your pipeline to access a feed that is in a different project. 2. You
    want your pipeline to be restricted from changing code in the repository. **3. You want your
    pipeline to be restricted from creating work items.**"
- URL (precise field names): https://learn.microsoft.com/azure/devops/pipelines/security/project-security-script?view=azure-devops#recommended-secure-by-default-settings
  - Quote: "`enforceJobAuthScope` | True | Restricts job access to only the current project."
  - Quote: "`enforceJobAuthScopeForReleases` | True | Same as above, for release pipelines."

**Answer to the precise sub-question** ("only repos/other-project, or can it also affect Boards
work-item read/query within the SAME project?"): The scope toggle's *primary documented effect*
is cross-project restriction (collection→project). Whether same-project Boards read/query is
affected is governed by the **project Build Service account's** Boards permissions, which include
area-path "View/Edit work items in this node" (see Q6). So: a project-scoped token CAN fail a
same-project work-item read/query **if** the `{Project} Build Service` identity lacks the area-path
"View work items in this node" permission (or has it Denied). This linkage is **A2 INFER** built
from two A1 facts (Q2 + Q6); the docs do not present it as one combined sentence. The "restricted
from creating work items" scenario is **A1** and confirms work-item operations are inside the
governed surface, not just repos.

## Q3 — az devops CLI auth precedence: does `AZURE_DEVOPS_EXT_PAT` get used in preference, and what is the precedence vs `az login`?

**VERDICT: CONFIRMED (A1), with precedence nuance.** When `AZURE_DEVOPS_EXT_PAT` is set AND you
have **not** signed in via `az login`/`az devops login`, the CLI uses the PAT. Official docs state
the precedence the other way for the conflict case: if you HAVE signed in interactively with
`az login`, you do **not** need to provide a PAT (the `az devops` commands use the `az login`
session). So the documented precedence is: an existing `az login` / `az devops login` credential
is used; the `AZURE_DEVOPS_EXT_PAT` env var is the path taken specifically when no such login
exists.

- URL: https://learn.microsoft.com/azure/devops/cli/log-in-via-pat?view=azure-devops#authenticate-with-the-azure%5C_devops%5C_ext%5C_pat-environment-variable
  - Quote: "For non-interactive or automation scenarios, set the `AZURE_DEVOPS_EXT_PAT`
    environment variable to your PAT. **If you haven't used `az login` or `az devops login`, all
    `az devops` commands attempt to use this variable for authentication.**"
- URL: https://learn.microsoft.com/azure/devops/cli/log-in-via-pat?view=azure-devops#sign-in-with-az-devops-login-and-enter-your-pat
  - Quote: "If you already signed in interactively with `az login`, you don't need to provide a
    PAT. The `az devops` commands support sign-in through `az login`."
- URL (pipeline pattern, corroborating that the env var IS the credential channel):
  https://learn.microsoft.com/azure/devops/cli/azure-devops-cli-in-yaml?view=azure-devops#authenticate-with-azure-devops
  - Quote: "You can authenticate using the System.AccessToken security token used by the running
    pipeline, by assigning it to an environment variable named `AZURE_DEVOPS_EXT_PAT`."

**Precise answer to "does the CLI use it as the credential in preference to an `az login`
token?"**: The official wording does NOT say `AZURE_DEVOPS_EXT_PAT` *overrides* an existing
`az login` token. It says the env var is used "if you haven't used az login or az devops login,"
and that an existing `az login` means you "don't need to provide a PAT." The first-party docs
therefore describe `az login` / `az devops login` as the established credential and the env var as
the fallback when none exists. A claim that the PAT env var takes strict precedence *over* a live
`az login` session is **NOT supported by official docs** (would be INFER[community] at best);
treat such a claim as unverified. Confidence on the documented behavior: high.

## Q4 — `az boards query` with cold cache, no `--org`/`--project`: git-remote auto-detect via `GET /_git/<repo>/vsts/info` cached in `~/.azure/azuredevops/cache/remotes.json`; and empty WIQL match = empty set vs error

This question has THREE distinct claims; they verify at different confidence levels.

### Q4a — Org/project auto-detected from git context when `--org`/`--project` omitted
**VERDICT: CONFIRMED (A1).** Every `az boards`/`az repos` command's reference documents that
`--org` and `--project` are "Required if not configured as default **or picked up via git
config**." So with no defaults and no flags, the CLI does attempt to resolve org/project from the
local git context. `az boards query` carries a `--detect` parameter ("Automatically detect
organization", default behavior) confirming auto-detection exists.

- URL: https://learn.microsoft.com/cli/azure/boards?view=azure-cli-latest  (`az boards query`)
  - Quote: "--org --organization … **Required if not configured as default or picked up via git
    config.**"
  - Quote: "--detect  Automatically detect organization. | Accepted values: false, true |"

### Q4b — Specific route `GET /_git/<repo>/vsts/info` and cache file `~/.azure/azuredevops/cache/remotes.json`
**VERDICT: INFER[community] / NOT IN OFFICIAL DOCS.** Microsoft's first-party docs describe THAT
detection happens (git config) and that a `--detect` flag exists, but do NOT document the
underlying HTTP route used for detection nor the on-disk cache path. The `vsts/info` endpoint and
the `~/.azure/azuredevops/cache/remotes.json` cache file are **implementation details** of the
`azure-devops` CLI extension; they are observable from the extension's source
(github.com/Azure/azure-devops-cli-extension) and community reports, not from learn.microsoft.com.
The org-URL-resolution *mechanism family* is corroborated by the Resource Areas / connection-data
docs below, but the exact `vsts/info` path + cache filename remain **INFER[community]**; verify by
reading the extension source or by `--debug` trace if FACT-grade certainty is needed.

- Supporting (mechanism family, A1, but NOT the exact route/cache):
  https://learn.microsoft.com/azure/devops/extend/develop/work-with-urls?view=azure-devops#get-an-organization's-url
  - Quote: "You can resolve an organization's base URL from its name or ID by using the global
    Resource Areas REST API (`https://dev.azure.com/_apis/resourceAreas`)."
- Non-first-party source for the exact route/cache (mark INFER[community]):
  azure-devops-cli-extension repository — `git config` remote parsing + `vsts/info` probe +
  `remotes.json` cache. (https://github.com/Azure/azure-devops-cli-extension — community/source,
  NOT learn.microsoft.com.)

### Q4c — A valid WIQL that matches nothing returns an empty set, not an error
**VERDICT: INFER (strong), NOT explicitly stated.** The Query By Wiql REST API (which `az boards
query --wiql` invokes) returns `200 OK` with a `WorkItemQueryResult` whose `workItems` is a
`WorkItemReference[]` array. The schema and the sample response model results as a list; a query
that matches nothing yields an empty `workItems` array (a successful 200, not a 4xx/5xx). Official
docs do NOT contain a sentence literally stating "no matches returns an empty array rather than an
error," so this is **A2 INFER** from the response schema (`workItems` is an array; matching is a
filter, not a precondition) — a syntactically valid query is a successful operation regardless of
match count. Confidence: high, but label as INFER since not verbatim-stated.

- URL: https://learn.microsoft.com/en-us/rest/api/azure/devops/wit/wiql/query-by-wiql?view=azure-devops-rest-7.1
  - Quote (Responses): "200 OK | WorkItemQueryResult | successful operation"
  - Quote (WorkItemQueryResult): "workItems | WorkItemReference[] | The work items returned by
    the query."

## Q5 — WIQL: does `System.Id IN ()` (empty IN list) or `IN ($emptyvar)` produce an ERROR or an empty result?

**VERDICT: INFER[unknown] — official docs DO NOT document empty-IN-list behavior.** The WIQL
syntax reference documents `IN` / `NOT IN` as supported operators for Integer/String/Identity/etc.
fields and shows non-empty examples (`IN ('Active','Approved',...)`), but it does NOT specify the
semantics or validity of an **empty** `IN ()` list or a variable that expands to empty. There is no
first-party statement that empty `IN ()` returns an empty set, nor that it errors.

- URL: https://learn.microsoft.com/azure/devops/boards/queries/wiql-syntax?view=azure-devops#specify-filter-clauses-%60where%60
  - Quote (operator support, non-empty only): "`Integer` | `… In, Not In, Was Ever`"
  - Quote (example, non-empty list): "`[System.State] IN ('Active', 'Approved', 'Committed', 'In Progress')`"
  - Note: NO example or statement covers `IN ()` with zero elements.

**Practical guidance (label INFER[community]):** community/observed behavior is that a literal
empty `IN ()` is typically a **WIQL parse error** (the grammar expects at least one value), whereas
the safe pattern in scripts is to guard against an empty id list before issuing the query (or use a
clause that cannot match, e.g. `System.Id = 0`). Because no first-party doc states this, treat
"empty IN () = error" as **INFER[community]**, not FACT. If the RCA's conclusion depends on whether
the script's empty-list path errored vs returned empty, that must be confirmed by reproducing the
exact WIQL against the org (runtime probe), not by docs.

## Q6 — Azure Boards: which permission governs SEE/query work items under an Area Path, and how area-path security is scoped

**VERDICT: CONFIRMED (A1).** The governing permissions are the **area-path object-level**
permissions **"View work items in this node"** (read/query) and **"Edit work items in this node"**
(modify). Area-path security is hierarchical and inheritable: it is set per classification node
(token `vstfs:///Classification/Node/{area_node_id}`), applies to the subtree, and a **Deny** on
"View work items in this node" overrides any implicit allow — even for administrators.

- URL: https://learn.microsoft.com/azure/devops/organizations/security/permissions?view=azure-devops#area-path-object-level
  - Quote: "View work items in this node `CSS, GENERIC_READ` Can view, but not change, work items
    in this area node."
  - Quote: "Edit work items in this node `CSS, WORK_ITEM_WRITE` Can edit work items in this area
    node."
  - Quote (Deny override): "If you set the **View work items in this node** to **Deny**, the user
    can't see any work items in this area node. A **Deny** overrides any implicit allow, even for
    users that are members of an administrative groups."
- URL: https://learn.microsoft.com/azure/devops/organizations/security/set-permissions-access-work-tracking?view=azure-devops#create-child-nodes,-modify-work-items-under-an-area-or-iteration-path
  - Quote: "Area path permissions let you manage access to edit or modify work items, test cases,
    or test plans assigned to those areas."
  - Quote (inheritance): "(Optional) Choose the **Inheritance** slider to disable inheritance.
    Disabling **Inheritance** persists all inherited permissions as explicit Access Control
    Entries (ACEs)."
- URL (token format / scoping): https://learn.microsoft.com/azure/devops/organizations/security/namespace-reference?view=azure-devops#object-level-namespaces-and-permissions
  - Quote: "Manages area path object-level permissions to create, edit, and delete child nodes and
    set permissions to view or edit work items in a node. … **Token format example:** `… "token":
    "vstfs:///Classification/Node/{area_node_id}" …`"
- URL (WIQL prereq ties query to this exact permission): https://learn.microsoft.com/azure/devops/boards/queries/wiql-syntax?view=azure-devops
  - Quote: "Permissions | **View work items** or **View work items in this node** permission set
    to **Allow**. These permissions are typically granted to members of the **Readers** and
    **Contributors** groups."

**Cross-link to Q2:** This is the permission that a project-scoped `{Project} Build Service ({Org})`
identity must hold (Allow, not Denied) for a pipeline's `System.AccessToken`-authenticated WIQL
query to see work items under a given area path. Combining Q2 + Q6 explains how a same-project WIQL
can legitimately return nothing (or fail authorization) without any WIQL syntax error.
