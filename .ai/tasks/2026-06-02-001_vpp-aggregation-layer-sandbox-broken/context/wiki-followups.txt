# Applications managed by ArgoCD
This document describes the configuration of ArgoCD on the Sandbox cluster and applications that
are managed through ArgoCD.

## 1. ArgoCD itself
Context: ArgoCD was installed based on the kubernetes manifests on ArgoCD's GitHub. After installation, the definitions were imported into ArgoCD itself.

|Installation method|ArgoCD application file|Kubernetes Namespace|External URL|
|-|-|-|-|
|Kustomize|[here](https://dev.azure.com/enecomanagedcloud/Myriad%20-%20VPP/_git/VPP.GitOps?path=/argocd-configuration/applications/argocd.application.yaml)|argocd|https://argocd.dev.vpp.eneco.com|

### Applied customizations:
#### Microsoft Azure Entra Login:
- OIDC configuration is patched into the argocd-cm configmap using Kustomize (see [argocd-cm.yaml](https://dev.azure.com/enecomanagedcloud/Myriad%20-%20VPP/_git/VPP.GitOps?path=/argocd/overlays/sandbox/argocd-cm.yaml))
- The clientsecret for the app registration is manually added to secret argocd-secret with key: oidc.azure.clientSecret
- The clientsecret is read in argocd-cm through a reference to the key in argocd-secret
- The clientsecret can be found in Azure Entra App Registration _appreg-vpp-argocd-id-d_ with name _argocd-configmap-argocd-secret-oidc-azure-clientSecret_

#### Authorization:
- Members of Azure Entra Security Group _sg-vpp-platform_ are assigned the _org-admin_ role through the configuration in [argocd-rbac-cm.yaml](https://dev.azure.com/enecomanagedcloud/Myriad%20-%20VPP/_git/VPP.GitOps?path=/argocd/overlays/sandbox/argocd-rbac-cm.yaml)

#### Ingress:
- URL: https://argocd.dev.vpp.eneco.com
- Ingress is handled through the Azure Application Gateway instance on sandbox
- The Ingress TLS certificate is loaded from Azure KeyVault through the Azure KeyVault CSI Driver
  - related files:
    - [secretproviderclass.yaml](https://dev.azure.com/enecomanagedcloud/Myriad%20-%20VPP/_git/VPP.GitOps?path=/argocd/base/secretproviderclass.yaml) defines the secretproviderclass for the CSI driver    
  - The patch in [argocd-server-secrets-store-volume.patch.yaml](https://dev.azure.com/enecomanagedcloud/Myriad%20-%20VPP/_git/VPP.GitOps?path=/argocd/base/argocd-server-secrets-store-volume.patch.yaml) patches a volume and a volumeMount into the _argocd-server_ deployment which triggers the CSI driver to load the secrets from Azure KeyVault and create the Kubernetes secret
  - The secret is referenced in [ingress.yaml](https://dev.azure.com/enecomanagedcloud/Myriad%20-%20VPP/_git/VPP.GitOps?path=/argocd/overlays/sandbox/ingress.yaml) which uses it as the TLS certificate for the ingress.
  - Since TLS is terminated by the Ingress Controller, the argocd-server runs in insecure mode, so it runs on HTTP inside the cluster. This is handled by the configmap [_argocd-cmd-params-cm_](https://dev.azure.com/enecomanagedcloud/Myriad%20-%20VPP/_git/VPP.GitOps?path=/argocd/overlays/sandbox/argocd-cmd-params.cm.yaml)
