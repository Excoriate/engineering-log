**Q4 — Hidden assumption in L4 SDK story.** rca.md L4 says "TWO independent validations at startup … (a) ValidateStorageConnectionAsync … (b) ValidateEventHubsConnectionAsync". Verified-diagnosis L4 row says the live log shows BOTH errors co-occurring; rca.md correctly cites both. **However**: Azure.Messaging.EventHubs `EventProcessorClient.StartProcessingAsync` does NOT call two named "Validate*" methods in parallel. The actual flow is: `StartProcessingAsync` → `EnsureLoadBalancerRunning` → `RunLoadBalancingAsync` → `ClaimOwnershipAsync` (which calls `BlobCheckpointStore.ListOwnershipAsync` → blob 404) AND independently `ProcessEventBatchAsync` per partition → `AmqpConsumer.ReceiveAsync` (AMQP open → ResourceNotFound). Both run in *concurrent partition tasks*, not as two validation calls in `StartProcessingAsync` itself. **For the fix's correctness this distinction is immaterial**: both Azure resources are still required to heal the pod; the IaC paired modules still create both atomically per (EH,CG); a single Apply still fixes it. But the wording "TWO independent validations at startup" is incorrect — the AMQP error fires from the receive loop after partition claim, not from a startup pre-flight validation. Patch 2 reflects this.
