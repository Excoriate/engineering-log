## MECH 1 — Work-item ID extraction — VERDICT: RESIDUAL (MEDIUM/LOW)

**Code (lines 49-50)**: `git log --format=%B | grep -F 'Related work items:' | grep -Po '\d+' | sort -u | paste -sd, - || true`

**Mechanism**: `grep -F 'Related work items:'` selects the WHOLE matching line **as an unanchored substring**, then `grep -Po '\d+'` extracts **every digit run anywhere on that line** — not just the `#NNN` ids.

**Decisive evidence** (harness `extract.sh` reproduces lines 49-50 with `ggrep`):

```
=== F: text AFTER ids on same line containing digits (HAZARD) ===
input:  Related work items: #123 (see also ticket 9999)
WORK_ITEMS=[123,9999]              <- stray 9999 captured

=== G: marker appears as SUBSTRING in unrelated prose ===
input:  I updated the Related work items: section formatting in 2024 for item 555
WORK_ITEMS=[2024,555]              <- fully spurious; no real work-item referenced

=== I: a DATE or version on the marker line ===
input:  Related work items: #123 fixed on 2026-06-26 v1.2
WORK_ITEMS=[06,1,123,2,2026,26]    <- date/version explode into garbage ids
```

Normal/clean ADO bodies are correct:
```
A: 'Related work items: #123'                  -> [123]
B: 'Related work items: #123, #456'            -> [123,456]
C: 'Related work items: #123, #124, #125.'     -> [123,124,125]   (trailing period harmless)
J: two commits each with marker                -> [100,200]
```

**Why it does not auto-escalate to BROKEN**: the spurious ids are only a CANDIDATE set; the WIQL adds `AND [System.Id] = 6393-area AND [System.Id] IN (...)`. A garbage number not present in BtM area 6393 returns nothing and tags nothing. The damage requires a stray number to **coincidentally equal a real BtM work-item id** — then that unrelated item gets the env tag (attack class (b), wrong tag).

**Counter-hypothesis**: *"ADO's autogenerated line is always clean `#NNN`, so stray digits never appear."* Rebuttal: (a) the match is unanchored `grep -F` (case G — the phrase appearing mid-prose harvests prose digits); (b) authors can edit the squash-merge commit body in the ADO merge dialog, adding free text on/after that line; (c) the area filter is the SOLE guard. I favor RESIDUAL over SURVIVES because the failure is real and the only mitigation is a coincidence-of-ids being unlikely — not impossible. Severity bounded to MEDIUM/LOW by the area filter.

**Conditional fix** (NOT applied — reporting only): anchor and scope the extraction to the `#NNN` tokens after the marker, e.g. capture only `#(\d+)` occurrences (`grep -Po '(?<=#)\d+'`) and/or restrict to the marker line's id-list segment. This removes stray dates/versions/prose digits.

## MECH 2 — WIQL `IN ($work_items)` validity + injection — VERDICT: SURVIVES

**Decisive evidence**:

```
O: WIQL INJECTION via commit body
input:  Related work items: #123') OR ([System.Id] > 0) --
WORK_ITEMS=[0,123]                 <- quotes/parens/-- STRIPPED; only digit runs survive
```
`grep -Po '\d+'` is a digits-only sieve: no quote, paren, or comment can reach the WIQL. **No injection via commit/PR text.**

IN-list shape:
```
single id   -> IN (123)     valid WIQL
paste -sd,  -> 1,2,3        no leading/trailing comma
real pipeline never emits blank lines: 'Related work items: #1\n\n\n#2' -> WORK_ITEMS=[1,2]
```
`paste` only produces leading/empty commas if fed blank lines (`,1,,2`), but `grep -Po` emits one match per matched run and never empty lines, so `sort -u` cannot introduce blanks. **IN-list always well-formed.**

A quote in `TAG` (`DEV' OR '1'='1`) WOULD inject — but `TAG` is operator-set in the pipeline YAML to `DEV`/`ACC`/`PRD`, never derived from PR/commit/branch. Requires pipeline-YAML write = already trusted. **SURVIVES** (noted as a latent surface only).

## MECH 3 — Tag UNION + idempotency exact-match — VERDICT: RESIDUAL (LOW)

**Code (lines 77-86)**. Skip iff `";$(tr -d ' ' <<<"$current");" == *";${TAG};"*`; else `new_tags="$current; $TAG"`.

**Truth table** (harness `union.sh`, verbatim logic):

| current | TAG | result | correct? |
|---------|-----|--------|----------|
| `` (empty) | DEV | WRITE `DEV` | yes |
| `None` | DEV | WRITE `DEV` | yes (null sentinel) |
| `DEV` | DEV | SKIP | yes |
| `ACC; DEV` | DEV | SKIP | yes (delimiter-bounded) |
| `DEV; ACC` | DEV | SKIP | yes |
| `predev` | DEV | WRITE `predev; DEV` | yes (substring, must NOT skip) |
| `DEV-stuff` | DEV | WRITE `DEV-stuff; DEV` | yes |
| `DEVELOPMENT` | DEV | WRITE `DEVELOPMENT; DEV` | yes |
| `ACC; PRD` | DEV | WRITE `ACC; PRD; DEV` | yes (union, no clobber) |

The exact-match skip is **correct**: it avoids the false "already has it" on `predev`/`DEV-stuff`/`DEVELOPMENT` (the precise hazard the brief asked about) by wrapping in `;...;` delimiters. Union appends without clobber.

**RESIDUAL found — `tr -d ' '` over-collapses internal spaces**:

```
D EV   | DEV -> SKIP (already has DEV)   <- FALSE SKIP: tag literally 'D EV' collapses to 'DEV'
DE V   | DEV -> SKIP (already has DEV)   <- FALSE SKIP
```
A tag whose non-space characters spell the env code (`D EV`, `AC C`, `P RD`) collapses to a false match and the item is **silently not tagged** (attack class (c)). ADO permits spaces in tag names. Probability is low (requires a bizarrely-named tag), hence **RESIDUAL/LOW**.

Secondary observation (case sensitivity, not counted): ADO tags are case-insensitive; `current="dev"`, `TAG="DEV"` does NOT skip and writes `dev; DEV`. ADO will likely dedupe on display, so this is cosmetic noise, not a clobber. Flagged for awareness.

**Counter-hypothesis**: *"tr -d ' ' is needed to normalize ADO's `A; B` spacing, and real env tags have no internal spaces so the collapse is irrelevant."* Partially true — `TAG` itself never has spaces; the collapse only misfires on a pre-existing tag that happens to spell the code with spaces. I keep RESIDUAL/LOW rather than dropping it because such tags are permitted and the failure is a silent no-tag.

**Conditional fix**: split on `;` and trim each element, compare element-wise (`[[ " ${arr[*]} " ...`]]`), instead of deleting all spaces globally.

## MECH 4 — `current` under `set -u` / null / show-failure — VERDICT: **BROKEN (HIGH)**

**Code (lines 75-77)**:
```bash
current=$(az boards work-item show "${wi_ctx[@]}" --id "$id" \
  --query "fields.\"System.Tags\"" -o tsv 2>/dev/null)
[[ "$current" == "None" ]] && current=""
```

**4a/4b SURVIVES**: under `set -u`, a failed command substitution yields a defined-empty string (not unset → no abort). Literal `None` from a genuinely empty field is correctly normalized to `""`.

**4d BROKEN — silent CLOBBER on the per-item read-failure branch (attack class (b))**:

The `show` call has `2>/dev/null` and **no failure guard** (unlike the `query`, which is wrapped in `if ! ids=$(...)`). `set -e` is OFF. So if `show` fails for any reason, `current=""`, the skip check is bypassed (current empty), and line 85 sets `new_tags="DEV"` — line 87 then `update --fields "System.Tags=DEV"` **replaces the entire tag set**.

**Decisive end-to-end evidence** (harness `e2e.sh`, full script logic, `az`/`git` stubbed; `SHOW_FAIL=1` simulates a failing `show` on an item genuinely tagged `ACC; PRD`):

```
================ CLOBBER PATH (4d): show FAILS, item really has ACC; PRD ================
WI 100: '' -> 'DEV'
  >> UPDATE id-fields: boards work-item update w --id 100 --fields System.Tags=DEV --query ... --output yamlc
exit=0
```

`ACC; PRD` are **gone**, replaced by `DEV`, and the run exits 0 with no warning — because the *update* succeeded; only the *read* failed, and that read failure was swallowed by `2>/dev/null`. This is **the exact bug the hardening claims to have fixed** (header line 22: "the original ... wrote `System.Tags=; $TAG`, CLOBBERING existing tags"). The union-read-first design eliminates it on the happy path but reintroduces it whenever the per-item `show` fails.

**Counter-hypotheses (tested, both rebutted)**:
- CH1 *"If query succeeded, show always succeeds (same identity), so unreachable."* — Rebutted: `query` (WIQL endpoint) and `show` (work-item GET) are **separate REST calls** with independent failure modes (429 throttling, transient 5xx, an item deleted/moved mid-run, partial outage). The script's OWN header (line 19) cites cold-cache/ephemeral-agent flakiness as a real, observed failure class — exactly this. Not unreachable.
- CH2 *"`--fields System.Tags=` merges, not replaces."* — Rebutted: `System.Tags` is a single delimited string field; the JSON-Patch replaces its value. The original clobber bug only makes sense under replace semantics, and the entire read-first union design presupposes replace. Internally consistent.

I favor BROKEN because the happy-path test shows correct union (`ACC -> ACC; DEV`) while the SHOW_FAIL test shows clobber (`ACC; PRD -> DEV`), isolating the read-failure branch as the cause. I would switch to RESIDUAL only if `az boards work-item show` could be proven to never fail independently of `query` — which CH1 refutes.

**Blast radius**: every work item whose `show` fails during a run loses ALL its existing tags, replaced by the single env tag — silently, exit 0. The query pre-filters to items NOT already containing `$TAG`, i.e. items that most likely DO carry other tags, maximizing the loss.

**Conditional fix** (NOT applied): guard the read; on `show` failure, `warn` and `continue` (skip the item) rather than proceeding with empty `current`. E.g. `if ! current=$(az ... show ...); then warn "could not read tags for $id; skipping to avoid clobber"; continue; fi` — and do NOT redirect `show` stderr to `/dev/null`, or capture it for the warning.
