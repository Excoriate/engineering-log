# Operational fix-soundness review

| Severity | Step | Flaw | Consequence | Correction |
|---|---|---|---|---|
| BLOCKING | 3 → 4 — update Key Vault, then restart consumers | The fix assumes Key Vault has already propagated to the Kubernetes Secret. The evidence explicitly leaves the KV→Kubernetes mechanism unproven (`E10`: CSI SecretProviderClass vs ESO). An environment variable is re-read at pod start, but only from the Kubernetes Secret version visible at that start; a restart does not itself prove that version contains the newly minted token. | Both rollouts can complete successfully while the new pods reload the old token and continue returning 401. This is the single most likely silent non-fix. | In the AVD, identify the deployed sync owner and exact Kubernetes Secret/key first. After updating Key Vault, force or wait for the mechanism-specific reconcile (CSI rotation/SecretProviderClass mount or ESO refresh), then prove the Kubernetes Secret changed using metadata plus a non-reversible digest comparison that never prints the token. Restart only after that gate passes. If the sync path cannot be proved, stop rather than guessing the order. |
| BLOCKING | 2 — “or repair the org/bucket if those were lost” | This phrase collapses two different routes. Minting a replacement token is reversible; recreating or “repairing” a missing org/bucket is stateful recovery with different IDs, retention, dashboards, token scopes, and possible data loss. It also weakens the otherwise-correct refusal to copy the sibling ticket’s delete/recreate action. | An operator can turn an authentication incident into a data-loss or dashboard-wide incident, or recreate an empty bucket that makes writes return 204 while historical monitoring remains lost. | Make this an explicit branch: if `vpp-agg` or `aggregation` is absent, HALT the token-rotation runbook. Capture instance/PV state, org and bucket inventory, backups, and recent change history; obtain Aggregation + Platform authorization for a separate recovery plan. Do not delete or recreate a collection, org, bucket, user, or InfluxDB instance as part of this fix. |
| HIGH | 2 → 4 — mint token and select consumers | “Any other writers that 401” is not a safe consumer inventory. `E10` says the other-writer blast radius is unproven. Updating a shared Key Vault secret can affect every deployment that references it on its next restart, including a writer that has not appeared in the retained App Insights window. | A token scoped for only the observed path can cause a delayed outage in another writer, or healthy consumers can remain on stale env values and fail later during an unrelated rollout. | Before cutover, enumerate every live Deployment/StatefulSet whose env or referenced Kubernetes Secret resolves to `influxdb-api-token`; record each org/bucket and required scope. Mint one write-only token that satisfies the proven shared contract, or separate credentials if contracts differ. Plan and verify every consumer, not only workloads with a currently visible 401. |
| HIGH | 4 — restart `strikepricefn` b2b and b2c | The evidence proves b2b/b2c variants exist in GitOps, not their exact live Deployment names, namespace bindings, Secret references, or current `oc` context. The step has no ambient-context guard and no sequencing/rollback checkpoint. | From the AVD, an operator can restart the wrong environment/workload, restart a variant that does not consume the rotated Secret, or roll both variants before learning that the first new pod is unhealthy. | Before mutation, capture `oc whoami --show-server`, current project, exact `eneco-vpp-agg` namespace, and label-selected live deployments. For each candidate, prove its env/secretKeyRef points to the refreshed Kubernetes Secret. Restart one deployment at a time with an exact name and namespace; require `oc rollout status`, Ready replicas, a fresh ReplicaSet/pod start time, and no startup/config error before moving to the next. All `oc` execution remains user-run in the AVD unless separately authorized. |
| HIGH | 5 — no new 401 + manual 204 + fresh Grafana data | The three signals can false-pass independently: no 401 can mean the 15-minute timer did not run or telemetry lagged; a manual 204 proves only the token used by the manual client, not the pod’s env; “fresh Grafana data” can come from another writer or an unrelated measurement through Grafana’s separate read token. | The incident can be declared resolved while one or both `strikepricefn` variants still use the stale token and continue dropping monitoring points. | Split verification into token and consumer gates. First, use the newly minted write-only token—not the admin or Grafana token—to write to the exact org/bucket and query back the uniquely timestamped point. Then, after each rollout, wait for at least one observed scheduled invocation after the pod start time; prove that invocation occurred, produced no InfluxDB exception, and advanced the exact strike-price measurement’s newest timestamp beyond the rollout. Attribute evidence per b2b/b2c pod or variant. Grafana is corroboration only. |
| MEDIUM | 1 → 3 — admin access and token cutover | The fix names the separate admin and Grafana credentials, but it does not state the safe token lifecycle. An operator could reuse the admin token as the writer, overwrite the Grafana read secret, mint all-access scope, expose a token in command history, or revoke an old token before the new path is proven. | The recovery can enlarge privilege, break Grafana, leak a credential, or remove the only remaining administrative recovery path. | Use the admin credential only to inspect and mint; never copy it into `influxdb-api-token`. Mint a side-by-side, named, write-only token for org `vpp-agg` and bucket `aggregation`; handle it without echoing it; update only `influxdb-api-token`. Leave `influxdb-admin-token`, `influxdb-admin-password`, and `grafana-influxdb-v2-influxql-datasource-api-token` untouched. Revoke an obsolete token by confirmed token ID only after all consumer effect gates pass. |

## Top 3 must-fix

1. Add a hard KV→Kubernetes reconciliation and changed-Secret proof before any rollout; restart alone can reload the old value.
2. Replace “repair the org/bucket” with a stop-and-escalate recovery branch; preserve the explicit ban on the sibling ticket’s delete/recreate action.
3. Inventory every consumer of `influxdb-api-token` and require a post-rollout, per-variant scheduled write that advances the affected measurement; manual 204 and generic Grafana freshness are insufficient.

**Verdict:** No — the causal fix shape is plausible, but it is not safe or sufficient until secret propagation, state-recovery branching, consumer scope, and end-to-end effect attribution are made explicit.
