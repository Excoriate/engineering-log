## Controls that survive review

- **One-way doors:** PASS. The runbook creates and write-tests a new write-scoped auth before KV promotion; org/bucket absence and dead admin credentials HALT; it never recreates data-bearing objects. Old-auth handling requires a unique `OLD_AUTH_ID`, uses reversible `auth inactive`, and leaves hashed/unidentifiable auths active. InfluxData documents `inactive` as reversible with `auth active`.
- **Azure environment pinning:** PASS for Azure resource operations. `$SUB` is present on every Key Vault and App Insights read/mutation, including cleanup. `az account set/show` establishes the ambient context; extension installation is local. Microsoft Learn confirms the used `secret set --file` and `secret download --id --file --overwrite` forms.
- **Secret basics:** PARTIAL. `set +o history`, `umask 077`, `read -rs`, `-o none`, stdin delivery, and `--file` avoid the obvious argv/terminal/history leaks. The residuals above prevent a full pass.
- **Tunnel cleanup primitive:** PARTIAL. `kill` plus `wait` is present, and firewall removal is conditional on `KV_RULE_ADDED`; neither tunnel establishment nor cleanup postconditions are verified.
- **Effect intent:** PARTIAL. The prose correctly says `invocations>0`, `unauthorized==0`, and empty=UNKNOWN, and excludes `runbook_probe`. The implemented queries are not sufficiently attributable or post-rollout-bounded to prove that claim.

## Verdict

**NO-GO as written for unattended copy/paste, and not yet safe enough to approve merely because the operator has judgment.** The asymmetric choices are good—diagnose before rotate, create before overwrite, never recreate, reversible inactivation—but three load-bearing mechanisms remain unsafe: wrong-cluster mutations bypass `ocx`, the consumer set is not closed over rollout/verification/revocation, and §8 can report success without proving the affected writer produced a post-rollout point. Fix the CRITICAL/HIGH items before execution; the MEDIUM items are required for the runbook's absolute secret-safety and fail-closed claims.

**Stakes-claim (HIGH):** executing this version can mutate the wrong cluster, inactivate a token still used by an omitted consumer, expose an admin password to an unverified local endpoint, or close while the target write is still failing.

**Falsifier:** a revised dry-run must show that every secret-bearing/mutating `oc` path invokes `ocx`; discovery and pre-revocation inventories are set-equal and include every controller kind; each writer has at least one Ready pod, an attributable post-rollout scheduled write, zero delayed InfluxDB 401s, and a post-rollout non-probe point; and cleanup proves the KV rollback/firewall/tunnel postconditions.
