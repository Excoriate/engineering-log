## Top-3 must-fix

1. **Eliminate the impossible/leaky token handoff (§6A→§7).** The shown standalone create plus argv-based Key Vault set guarantees either disclosure or placeholder corruption across the split AVD/Azure planes.
2. **Mechanically pin dev-mc at every mutation.** The current `expect` comments and ambient Azure/OpenShift contexts do not prevent acc/prd actions.
3. **Remove autonomous `auth delete`.** The old authorization ID is not uniquely proven and §8 cannot prove every consumer; use reversible inactivation only after per-consumer evidence, then defer deletion to an explicitly authorized post-soak action.

## Genuinely safe gates to preserve

- The org/bucket-missing and dual-admin-auth-failure STOP branches occur before token creation and correctly prohibit stateful recreation.
- The proxy-origin STOP occurs before token rotation and correctly prevents fixing the wrong authentication layer.
- Hash-only KV → Kubernetes Secret → pod comparisons avoid printing the application token value.
- Enumerating consumers before rotation, creating a least-privilege token side-by-side, serial rollout status checks, and requiring an observed write are correct safety principles. They are not yet mechanically complete enough to authorize revocation.
- Cleanup verifies whether the firewall rule remains, but that check must move into an unconditional finally path.

## External safety confirmations

- [Azure CLI Key Vault secret set](https://learn.microsoft.com/cli/azure/keyvault/secret?view=azure-cli-latest#az-keyvault-secret-set) supports `--file` and per-command `--subscription`; `--value` is plaintext argv and cannot be combined with `--file`.
- [InfluxDB token creation](https://docs.influxdata.com/influxdb/v2/admin/tokens/create-token/) exposes plaintext only immediately after creation, so a logged standalone create cannot be repaired later without another rotation.
- [InfluxDB auth inactive](https://docs.influxdata.com/influxdb/v2/reference/cli/influx/auth/inactive/) is the reversible alternative to deletion; the vendor explicitly recommends inactivation for temporary disablement.

## Verdict

**NOT-READY**

The runbook has correct high-level ordering, but a literal one-shot agent can leak both write and administrator tokens, mutate the wrong environment, leave the Key Vault firewall open, corrupt the promoted secret after an earlier command fails, restart unsafe/unbounded scope, and irreversibly delete the wrong authorization. The STOP prose does not mechanically block those paths.
