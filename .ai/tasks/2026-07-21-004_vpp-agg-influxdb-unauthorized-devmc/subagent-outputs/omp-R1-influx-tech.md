| severity | location | problem | why | fix |
|---|---|---|---|---|
| HIGH | `explanation.md:74-81`, `:168-177`, `:218`, `:226` | The high-level distinction—401 for rejected authentication, 403 for an authenticated token lacking permission—is useful, but the explainer over-promotes a 401 into proof that the token record is absent or “unknown to the instance.” A 401 proves that InfluxDB did not accept the presented credential as valid authentication. It does not distinguish a nonexistent token from a known authorization that is inactive/revoked, a deactivated owner, or other authentication-state invalidation. | “Unknown token” is the load-bearing step used to rule out every mechanism except orphaning. An operator could stop investigating authorization status or identity state even though the response alone does not make that distinction. | Keep the 401/403 table, but replace “I don't recognise this token at all” and “unknown to the current instance” with “the presented token was not accepted as valid authentication.” State that deleted/orphaned token is the leading hypothesis, while inactive authorization or deactivated ownership remain alternatives until the live authorization record is checked. |
| HIGH | `explanation.md:71-73`, `:175-179`, `:195-198` | Token identity, resource scope, and instance lifecycle are conflated. Restarting or rebuilding the InfluxDB workload with the same persistent metadata does not invalidate tokens. Recreating only a bucket can leave the token recognizable while its bucket-ID permission targets the old resource, which is an authorization problem rather than proof of an unknown token. Only deletion/inactivation of the authorization or loss/reinitialization of the metadata that contains authorizations reliably makes the stored token unusable as authentication. | The proposed causal link to “collection” recreation is too broad. It can send responders toward server reinitialization when the actual distinction is authorization-record loss versus permission/resource-ID drift. | Split the mechanisms explicitly: (1) authorization deleted/inactivated or metadata reset → credential rejected, typically 401; (2) token still valid but permission references the wrong/new bucket → permission failure, typically 403; (3) pod/container restart with persistent metadata → token remains valid. Probe authorization status and current org/bucket IDs before naming the cause. |
| HIGH | `explanation.md:9`, `:173-189`, `:195` | The opening and fix section state the token is orphaned and that this is “not a code or network problem,” while the same document later labels the mechanism A2 and says the decisive InfluxDB-internal checks are blocked. A 401 proves that an HTTP endpoint responded; it does not by itself prove that the client reached the intended InfluxDB instance, nor that host/org/bucket configuration was unchanged when failures began. | The zero-context reader is taught to treat an unconfirmed leading hypothesis as established root cause. A wrong host or newly selected instance can produce the same rejected-token symptom and changes the remediation. | Make the one-sentence answer: “InfluxDB is rejecting the presented write credential; the leading hypothesis is an orphaned/deleted/inactive authorization, pending the in-instance check.” Remove the categorical exclusion of code/network/topology until the live endpoint identity and configuration history are verified. Change “Because the fault is” to “If the in-instance probe confirms an orphaned write token.” |

A separate token isolates a writer-token-only deletion or rotation, but the explainer also proposes org/user/whole-instance reinitialization; those mechanisms can invalidate Grafana's authorization or remove the data resources it reads. | This can understate blast radius and cause responders to skip checking the read path precisely when the leading server-side-reset hypothesis predicts broader damage. | Say “Grafana uses a separate token, so writer-token-only failure does not directly invalidate it; org/instance reset could still affect it.” Add a Grafana query or its token status to the in-AVD blast-radius checks. |
| LOW | `explanation.md:61` | “Every API call must carry a token” is false literally. ## Top 3 must-fix

1. Stop treating 401 as proof that no token record exists; it proves rejected authentication, while deletion, inactivation, and owner/auth-state invalidation require the live authorization check to distinguish.
2. Separate authorization-record loss from bucket/org resource-scope drift and from a harmless workload rebuild using persistent metadata.
3. Fix the displayed line protocol so the document does not teach an invalid wire payload.

**Verdict:** No — the 1.x/2.x model, write path, and self-managed OSS 2.x “API tokens do not expire” point are broadly correct, but the 401 inference and resulting root-cause certainty are too strong for a zero-context engineer.
