## Branches that are genuinely closed

- The proxy-issued-401 branch (§5b) terminates in STOP/escalation and correctly forbids token mutation, once response classification is made executable.
- The no-admin-auth or missing-org/bucket branch (§5c/§6C) clearly halts and forbids recreating stateful objects. That is the correct terminal policy; only universal firewall/port-forward cleanup is missing.
- The valid-and-write-scoped-token branch (§5c) routes back to byte/proxy checks and then escalation rather than forcing rotation. The policy is complete, although the current auth-ID matching procedure cannot establish that state safely.
- Create-before-revoke and “revoke only after every consumer is green” are correct safety invariants. Their implementation is not complete because the consumer ledger and per-writer proof are not executable.

## Top-3 must-fix

1. **Make credential movement executable across the stated Azure/AVD boundary.** Fix `ADMIN` lifetime, the admin-password branch, and the hidden transfer of the newly minted token into Key Vault. As written, the main repair cannot run without violating secret handling.
2. **Replace the hash-mismatch shortcut with a real decision matrix.** A mismatch does not prove KV is correct and must not bypass proxy/auth validation. Establish which bytes the live InfluxDB accepts before changing delivery.
3. **Make “fixed” a canonical-writer, pod-originated effect gate.** Persist one full writer set through §§4/7/8/9, exclude the §5b manual probe, map new pods after rollout, and require a scheduled write attributable to every writer before revocation.

## Verdict

**NOT-READY**

The runbook contains multiple reachable dead-ends and a direct false-pass: the §5b manual point can satisfy §8b while non-strikeprice writers are neither queried nor attributable. It cannot be executed one-shot across the explicitly separate Azure and AVD planes, and it has no rollback route for a failed rollout.

**Stakes-claim:** As written, an executor can be stranded before token creation or can revoke the old authorization without proving that every InfluxDB writer produced a post-rollout, pod-originated scheduled write.

**Falsifier:** Dry-run the revised graph with separate Azure and AVD shells, a canonical set containing the three strikeprice deployments plus `telemetryingestionfn`, and injected outcomes for hash mismatch, admin-token-401/password-success, rollout timeout, and one still-broken writer. This review is disproven only if every outcome has an executable successor and §8 remains red until each exact writer emits attributable post-rollout data.

**Stakes-class:** HIGH
