# Completeness review

## Findings

### Decision-tree trace

- **§2:** every failed assertion exits; success reaches §3. Closed.
- **§3:** active fault reaches §4. Zero-result policy says do not mutate, but its proposed §8 confirmation is not executable as a success gate; not closed.
- **§5a:** all hashes equal reaches §5b. Any divergence reaches §6B, but §6B neither determines which copy the server accepts nor provides executable recovery; not closed.
- **§5b:** `200`, InfluxDB `401`, proxy `401`, and other/non-HTTP outcomes all have stated successors. Closed as routing, although `200` deliberately escalates rather than repairs.
- **§5c:** admin failure reaches the password branch; missing org/bucket stops. The admin-success absent/inactive-versus-valid/write-scoped decision is not implemented by the script; not closed.
- **§6:** 6A reaches §7; 6C stops. 6B is prose-only and can declare done without §8; not closed.
- **§7:** full success reaches §8. The first rollout-status failure attempts only a KV restore; second-rollout failure, hash mismatch, or any other post-KV error bypasses recovery; not closed.
- **§8:** pass/fail/unknown are described but never parsed or enforced. The shell can continue directly into §9 after UNKNOWN or FAIL; not closed.
- **§9:** known `OLD_AUTH_ID` inactivates and unknown leaves it active. The policy is closed, but the known-ID state cannot be established by the current §5c code.

## Prior gaps now closed

- **Single execution plane:** the runbook now explicitly requires one AVD WSL shell where both `az` and `oc` work (§0 lines 16–17); the former cross-terminal variable/secret dead-end is closed.
- **Universal local cleanup:** the EXIT trap now owns the port-forward PID, only the firewall rule added by this run, temporary token file, and secret variables (§0 lines 18–37). The prior cleanup-only-on-success gap is closed.
- **Environment fail-closed preflight:** subscription, cluster, namespace, permissions, and Key Vault access have concrete assertions and STOP exits (§2).
- **401-origin routing:** `200`, InfluxDB `401`, proxy/mesh `401`, and unexpected statuses now have explicit successors (§5b).
- **Missing org/bucket and no-admin-auth terminals:** both halt and forbid recreation (§5c/§6C); those policy dead-ends are closed.
- **Org/bucket non-empty checks:** the admin-token path now asserts both identifiers before token creation (§5c lines 218–221).
- **Port-forward ownership:** the password fallback captures `PF_PID`, and the trap removes it (§5c-pw plus §0).
- **Diagnostic probe identity:** the new-token write uses a unique `runbook_probe` measurement and incident tag (§6A). This closes the untagged-probe gap, but not the §8 attribution/exclusion gap above.
- **Post-rollout pod rediscovery:** §8 resolves current Running pod names rather than reusing pre-rollout instances.
- **Ambiguous old auth policy:** §9 leaves an unknown `OLD_AUTH_ID` active and escalates instead of guessing or deleting.

Not counted as closed: full consumer discovery, §6B execution, auth-ID matching, password-fallback verification, enforceable empty=UNKNOWN handling, per-writer write attribution, and complete rollback.

## Verdict

**NOT-READY**

The normal happy path is more self-contained, but the one-shot contract still fails on four required fault injections: a non-Deployment writer, any §5a divergence, admin-token-dead/password-success, and a rollout/hash failure after KV mutation. Most importantly, §8 can reach §9 without machine-enforced proof for every writer, while §4 omits entire workload kinds. The runbook must not authorize old-token inactivation until those paths are closed.
