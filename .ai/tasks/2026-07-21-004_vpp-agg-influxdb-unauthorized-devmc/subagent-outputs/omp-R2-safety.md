# R2 safety review

## Findings

| ID | Severity | Exact step | Harmful scenario | Required correction |
|---|---|---|---|---|
| R2-S01 | CRITICAL | §5c lines 212–225; §6A lines 246–248; §8 line 301; §9 line 312 | `export INFLUX_TOKEN=…` puts the admin token in the environment of every later child process. More directly, `oc … -- env INFLUX_TOKEN="$INFLUX_TOKEN" influx …` expands the token into the local AVD `oc` process argv and into the remote `env` command argv; the OpenShift exec request also carries that command argument. `env` later replaces itself with `influx`, but the exposure already occurred. | Keep the admin token unexported locally. Feed it through stdin to `oc exec -i`; have a static remote `sh -c` perform `IFS= read -r INFLUX_TOKEN`, export it only for `influx`, then unset it. Never put `KEY=$SECRET` after `oc exec --`. Unset the local variable immediately after the final admin operation. |
| R2-S02 | CRITICAL | §7 rollback, line 276 | The rollback command substitutes the previous application token into `az keyvault secret set --value …`. The old write token is therefore present in the local `az` argv, process inspection, and command/audit capture. This directly violates the runbook's line-52 prohibition. | Before promotion, copy the previous secret version into a mode-600 temporary file under `umask 077`; restore with `az keyvault secret set … --file "$TMPOLD"`. Remove the file in unconditional cleanup. Do not use command substitution inside `--value`. |
| R2-S03 | CRITICAL | §2b line 133 versus every later `oc exec`, `port-forward`, `rollout restart`, and `auth inactive` mutation | The dev API is asserted once, but all later OpenShift operations use ambient kubeconfig state. Another AVD shell can change the shared current context after preflight. The same namespace/workload names can then resolve in acc/prd, allowing token creation, rollout, or inactivation there. Azure resource calls are individually pinned with `--subscription "$SUB"`; OpenShift mutations are not equivalently pinned. | Capture a dev context and route every `oc` command through a wrapper that uses that context/server and rechecks `oc … whoami --show-server == "$DEV_API"` immediately before each mutation. Prefer an explicit `--context`/`--server` on every operation. A one-time assertion is insufficient. |
| R2-S04 | HIGH | §5c lines 223–229; §9 lines 310–316 | `OLD_AUTH_ID` is not bijectively derived. The first jq expression errors on representative token JSON; the second uses unsupported `@sha256` and also errors. Both errors are suppressed, while §9 tests only non-emptiness, not exactly one valid authorization ID. The run can abort after a successful rollout or attempt inactivation with malformed data; the promised identity gate is not executable. | Hash retrievable tokens without printing them, collect all matching IDs, and set `OLD_AUTH_ID` only when the match count is exactly one and the ID matches the expected ID format. If tokens are redacted/hashed or match count is not one, leave it unset and skip inactivation. Do not suppress matcher errors. |
| R2-S05 | CRITICAL | §4b lines 171–176; §7 lines 273–284; §8 lines 289–305; §9 lines 310–312 | Rollout and verification iterate the same `${WRITERS[@]}`, which is good, but discovery asserts only `strikepricefn`; it does not require the known `telemetryingestionfn` consumer. It scans Deployments only. Verification selects at most one pod, its owner-reference fallback cannot map a Pod directly to a Deployment because the owner is normally a ReplicaSet, it does not assert `NEWPOD`, and the App Insights query merely prints a table. `total=count()` counts exceptions, not successful scheduled invocations; the bucket query is shared and not writer-attributed. The script then reaches §9 without a machine-set all-writers-green predicate, so an old auth can be inactivated while hidden or unexercised writers still use it. | Assert all known required writers, including `telemetryingestionfn`, are present or explicitly dispositioned; discover every supported workload kind and exact secretKeyRef/env mapping. Capture each new pod UID/name during rollout, assert non-empty and all replicas covered, and parse JSON evidence into `ALL_WRITERS_GREEN=1` only after each writer has an attributed successful post-rollout write and zero exact InfluxDB 401s. Require both `ALL_WRITERS_GREEN=1` and a bijective `OLD_AUTH_ID` in §9. |
| R2-S06 | HIGH | §7 lines 264–284 and all later failure exits | Rollback runs only when the first `rollout status` command fails. A second hash mismatch, later writer failure, verification failure, interrupt, or shell termination leaves the new KV version promoted. The rollback neither re-rolls already changed writers onto the restored value nor inactivates the newly created authorization, so partial state and an orphan active token remain. The rollback itself currently leaks the old token per R2-S02. | Add transaction state to the ERR/EXIT path: before final success, restore KV through a secure file on every failed promotion/rollout/verification path, roll already changed writers back or explicitly preserve both active tokens and halt before inactivation, and inactivate the new authorization on abort when its ID is known. Record and surface rollback failure; never swallow it. |
| R2-S07 | HIGH | Cleanup lines 24–31; firewall add lines 142–149; port-forward line 234; finish line 317 | The EXIT trap and `PF_PID` capture are improvements, but the script never checks whether `ADDED_IP` was already present before `network-rule add`. An idempotent add can set `KV_RULE_ADDED=1`, after which cleanup removes another operator's pre-existing rule. Removal errors are swallowed and absence is never verified. Because the runbook is multiple blocks pasted into one interactive shell, EXIT cleanup may not run when §9 finishes; it waits for the whole shell to exit. `kill` is not followed by `wait`. | Query the exact IP rule before add; set the ownership flag only when this run changed absent→present; verify presence after add and absence after remove. Treat cleanup failure as blocking. Execute the procedure as an actual script/subshell or call `cleanup; trap - EXIT` explicitly at the end. Bind port-forward explicitly to loopback, then `kill` and `wait` for it. |
| R2-S08 | CRITICAL | Branch B §6B, lines 255–257 | On any KV/Kubernetes/pod hash divergence, Branch B tells the executor to roll all writers but provides no executable probe that identifies which copy InfluxDB accepts. If the running pod/Kubernetes token is valid and the KV version is bad, a rollout propagates the bad KV value to every writer and expands the outage. The prose says §6B tests the copies, but no such commands exist. | Before any Branch-B rollout, test each candidate token through stdin against the same non-committing InfluxDB auth probe without printing it. Roll only after the accepted source of truth is proven. If no candidate is uniquely accepted, halt; do not restart the writer set. |
| R2-S09 | HIGH | §5c-pw lines 231–237 | The password fallback is neither autonomous nor secret-transfer-complete. It requires browser/UI interaction, gives no non-printing method to retrieve the Key Vault admin password into that UI, and asks the agent to capture the UI-created token later with `read -rs`. A context-free autonomous executor must improvise via terminal output, clipboard, or transcript, reopening the prior admin-secret leak. | Make this an explicit human-gated STOP branch, or provide a fully specified hidden, non-logged transfer mechanism. Do not describe the overall run as autonomous while this branch requires UI judgment and manual secret movement. |

Secret-hygiene non-findings: `set +o history`, `umask 077`, `az keyvault secret set --file`, `read -rs`, the temporary new-token file, and the new-token stdin write probe all move in the correct direction. The stdin probe does not place `NEW_TOKEN` in argv, although `printf '%s\n'` plus `IFS= read -r` is safer than `echo` plus plain `read`. Any token consumed through `INFLUX_TOKEN` remains visible to same-UID/root process-environment inspection inside the pod; that is a bounded runtime trust boundary, unlike the current local/remote argv exposure.

## Prior CRITICAL disposition

Closed:

- **V4-01 — new write-token handoff:** closed for the primary path. Token creation stdout is captured, promotion uses a mode-600 temporary file plus `--file`, and the new-token proof uses stdin rather than argv.
- **V4-04 — strict one-shell failure handling/placeholders:** closed. Strict mode, concrete variables, non-empty checks, and bounded second rollout replaced the unset-variable and literal-placeholder path.

Not fully closed:

- **V4-02:** open — admin token is still globally exported and passed in local/remote argv; the password branch is not safely specified.
- **V4-03:** open — Azure resource commands are pinned, but OpenShift mutations can drift to acc/prd after the one-time dev assertion.
- **V4-07:** partial — irreversible `auth delete` is correctly replaced by reversible `auth inactive`, but unique old-ID derivation and the all-writers-green gate are not implemented.
- **V4-08:** open — the same writer array is looped, but complete discovery and per-writer exercised-success proof remain non-enforced.

## Verdict

**NOT-READY**

A literal one-shot executor can still expose the admin or rollback token in argv, act on a drifted acc/prd OpenShift context, propagate a bad KV token across all discovered writers, remove a pre-existing firewall rule, and inactivate the old authorization without executable proof that every writer is green. These are active CRITICAL paths, not documentation polish.
