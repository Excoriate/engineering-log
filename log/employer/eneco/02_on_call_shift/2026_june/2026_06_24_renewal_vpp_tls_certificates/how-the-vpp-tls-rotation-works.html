How the VPP *.vpp.eneco.com TLS Rotation Works — and Why Only Production

Status: review · Author: Alex Torres (paired with Claude) · 2026-06-24 · Audience: VPP on-call / platform engineer with no prior context on this cert

This is the understanding companion to rotation-execution-spec.md. The spec tells you what to run; this tells you why it works, so you (or the next on-call) can reason about it, spot a wrong turn, and defend the scope under questioning.

Knowledge Contract

  2. explain why we import a new version of an existing Key Vault object (not a new object), and why a versionless secret URI makes that safe;

TL;DR

We received a renewed wildcard certificate for *.vpp.eneco.com. Live inspection proved it renews exactly one object — wildcard-vpp-eneco-com in the production Key Vault vpp-appsec-p — whose current cert expires 2026-07-01. App Gateway vpp-ag-p reads that object through a versionless link, so we add a new version, force one refresh, and four production sub-domains serve the new cert. dev and acc use different certificates and are untouched.

How we know this — the live inspections

The TL;DR says "live inspection proved." Here is exactly what was run, why, and what came back — nothing is asserted from memory or the runbook. Each probe answers one question and feeds the next.

1. What certificate do we actually hold? (local, no Azure)

Rationale: everything downstream depends on the cert's identity (its SAN names + validity). The PFX is password-protected and legacy-encrypted, so OpenSSL needs -legacy and reads the password from the file (never echoed).

openssl pkcs12 -in 26061584690-_-vpp-eneco-com.pfx -nokeys -clcerts \
  -passin file:certificate_password.txt -legacy \
  | openssl x509 -noout -subject -issuer -enddate -ext subjectAltName

Found: subject=CN=*.vpp.eneco.com; SAN *.vpp.eneco.com, vpp.eneco.com; issuer Trust Provider B.V.; notAfter=Dec 30 2026. The private key was confirmed to match the leaf — a complete, importable key-pair.

2. Which App Gateway and which Key Vault object? (control-plane, no firewall change)

Rationale: the colleague's runbook gave three different gateway names, so names had to be resolved live. The gateway's SSL-cert config carries the keyVaultSecretId, which reveals the exact object each listener binds — readable without opening any firewall.

az network application-gateway list -g mcprd-rg-vpp-p-res --subscription <prod> --query "[].name" -o tsv
az network application-gateway ssl-cert list --gateway-name vpp-ag-p -g mcprd-rg-vpp-p-res \
  --subscription <prod> --query "[].{name:name, kvSecretId:keyVaultSecretId}" -o json
az network application-gateway http-listener list --gateway-name vpp-ag-p -g mcprd-rg-vpp-p-res \
  --subscription <prod> --query "[?sslCertificate!=null].{host:hostName, sslCertId:sslCertificate.id}" -o json

Found: the real gateway is vpp-ag-p (the runbook's vpp-agw-p/vpp-appgw-p were both wrong). Its listeners agg/gurobi/apollo/flex-trade-optimizer.vpp.eneco.com bind ssl-cert wildcard-vpp-frontend-https, whose keyVaultSecretId is the versionless URI …/secrets/wildcard-vpp-eneco-com. So the object to rotate is wildcard-vpp-eneco-com.

3. Does the object actually hold our cert? (data-plane — needs a brief, surgical whitelist)

Rationale: names are not proof. The only certainty is reading the cert bytes the object holds today and comparing to the PFX. The KV firewall denies by default, so the operator IP is whitelisted only for the read, then removed.

az keyvault network-rule add --name vpp-appsec-p -g mcprd-rg-vpp-p-res --subscription <prod> --ip-address <myip>/32
sleep 25
az keyvault certificate show --vault-name vpp-appsec-p --name wildcard-vpp-eneco-com --query cer -o tsv \
  | openssl base64 -d -A | openssl x509 -noout -subject -enddate -ext subjectAltName
az keyvault network-rule remove --name vpp-appsec-p -g mcprd-rg-vpp-p-res --subscription <prod> --ip-address <myip>/32

Found (prod): CN=*.vpp.eneco.com, SAN *.vpp.eneco.com, vpp.eneco.com, notAfter=Jul 1 2026 — same names + CA as the held PFX, expiring imminently, i.e. the old version of what we hold. The same reads on dev and acc returned dev-mc.vpp.eneco.com/*.dev-mc.vpp.eneco.com and acc.vpp.eneco.com/*.acc.vpp.eneco.com — different certs. After every read the whitelist was removed and residual rules re-checked as zero.

4. Who can do the rotation? (identity + permissions)

Rationale: the actor needs two different permissions — write the KV firewall (control-plane) and import a certificate (data-plane access policy). Your own account lacks acc/prod rights; the MC service principal has both.

# As the human user, on acc/prd:
az keyvault network-rule add --name vpp-appsec-p ...   # → AuthorizationFailed (no Microsoft.KeyVault/vaults/write)
# As the prd service principal (cached from 1Password):
az keyvault network-rule add --name vpp-appsec-p ...   # → succeeds
az keyvault show -n vpp-appsec-p -g mcprd-rg-vpp-p-res --subscription <prod> \
  --query "properties.accessPolicies[?objectId=='<sp-object-id>'].permissions.certificates"

Found: the human user has no active role on acc/prd (hence the SP workaround). The prd SP's access policy includes Get, List, Create, Import, Update, Delete on certificates — so it can both whitelist and import.

5. Is the App Gateway the only thing using this cert in production?

Rationale: a wildcard could be consumed elsewhere (another gateway, Front Door, API Management, AKS ingress). Any such consumer left on the old cert would break at the Jul 1 expiry, so the whole production subscription was swept.

az network application-gateway list --subscription <prod> --query "[].{name:name,rg:resourceGroup}" -o table
az graph query -q "resources | where subscriptionId=='<prod>' | where type in~ (
  'microsoft.network/frontdoors','microsoft.cdn/profiles','microsoft.apimanagement/service',
  'microsoft.web/sites','microsoft.containerservice/managedclusters') | project type,name,resourceGroup"

Found: exactly one App Gateway in the prod subscription (vpp-ag-p) and zero Front Door / CDN / API Management / App Service / AKS resources. So within production this cert has a single consumer.

6. Will a manual import fight terraform? (drift check)

Rationale: if the cert object or its gateway binding is terraform-managed, a manual import could be reverted by CI or cause plan drift.

rg -l --glob '*.tf' 'azurerm_key_vault_certificate' /path/to/eneco-src          # → no matches
rg 'wildcard-vpp-eneco-com' /path/to/MC-VPP-Infrastructure/main/configuration/prd.tfvars

Found: no azurerm_key_vault_certificate resource anywhere — the cert object is not terraform-managed. The gateway binding is in terraform but uses the versionless URI, so importing a new version leaves the configured value unchanged → no drift; and the prod pipeline is trigger: none (no surprise auto-apply).

7. How does propagation actually work? (Microsoft documentation)

Rationale: the draft assumed an empty az network application-gateway update would force a refresh. That had to be confirmed — if wrong, the new cert would sit unserved until expiry.

Found (Microsoft Learn, App Gateway ↔ Key Vault troubleshooting, "Resolution E"): "Application Gateway refetches the certificate from Key Vault only when the configured keyVaultSecretId changes. … running an empty az network application-gateway update command … doesn't force the gateway to pull the newer Key Vault version." So the documented force-refresh points the binding at the versioned URI and then back to versionless — exactly what the spec does.

These seven steps are the chain behind every claim in this document and the spec.

First principles (the ladder)

Before any command, six small truths the change rests on:

Term            → what it is
  TLS cert      → public key + the names it is valid for, signed by a CA; proves "I am *.vpp.eneco.com"
  CN            → Common Name: the cert's single primary name (legacy). Here: *.vpp.eneco.com
  SAN           → Subject Alternative Name: the FULL list of names/wildcards a cert is valid for.
                  Modern browsers validate the hostname against the SAN list (CN alone is ignored).
                  OUR cert's SAN = { *.vpp.eneco.com , vpp.eneco.com } — the field we compared in Key Vault.
  Wildcard      → a SAN entry like *.vpp.eneco.com that matches EXACTLY ONE left label (host.vpp.eneco.com)
  PFX (PKCS#12) → one file: leaf cert + chain + PRIVATE key, password-protected
Primitive       → what must exist first
  Key Vault     → stores the cert as an OBJECT that holds many VERSIONS over time
  Secret URI    → how a consumer references it; "versionless" = no version GUID on the end
Invariant       → what can't change without changing the thing
  AGW binding   → the listener points at the versionless URI → "latest enabled version" wins
Mechanism       → how they interact over time
  Rotation      → add a new VERSION under the same object → versionless consumers pick it up
Consequence     → what the user sees
  Served cert   → the handshake on host.vpp.eneco.com presents whatever AGW last pulled
Failure         → what breaks an invariant
  Stale serve   → if AGW never re-pulls, it serves the OLD version until expiry → TLS errors

The key line is the wildcard one: *.vpp.eneco.com matches agg.vpp.eneco.com (one label) but not apollo.acc.vpp.eneco.com (two labels). That single fact is the whole "why only production" argument.

The chain the certificate travels, from the Networking4All email to the browser handshake — read it as "who hands the cert to whom."

What you are looking at: the PFX flows into one Key Vault object, which accumulates versions. The App Gateway does not point at a version — it points at the object through a versionless link (dotted). So the moment a new version becomes latest-enabled, every downstream listener is eligible to serve it. All four listeners share the same AGW ssl-cert resource (wildcard-vpp-frontend-https) — which is why one rotation covers all four, and why a check on any one is evidence for all. Keep this picture: object holds versions; the consumer references the object, not a version.

The two-name trap: the KV object is wildcard-vpp-eneco-com, but the AGW ssl-cert resource referencing it is wildcard-vpp-frontend-https. Different names for different things — commands that confuse them fail with "not found."

The mechanism over time — what happens, in order

Topology shows what exists; this shows causality — the order that makes it safe, and the one place it silently fails.

sequenceDiagram
  participant Op as Operator (prd SP)
  participant KV as Key Vault object
  participant AGW as App Gateway
  participant Cli as Browser
  Op->>KV: import new version (DISABLED)
  Note over KV: exists but not "latest enabled"
  Op->>KV: read new thumbprint
  Op->>Op: compare to held PFX (gate)
  Op->>KV: enable new version
  Note over KV: versionless URI now resolves to NEW
  Op->>AGW: repoint ssl-cert to NEW versioned URI
  Note over AGW: a keyVaultSecretId CHANGE forces re-pull
  AGW->>KV: fetch NEW cert
  Op->>AGW: restore versionless URI (keeps autorotation + matches terraform)
  Cli->>AGW: TLS handshake
  AGW-->>Cli: presents NEW cert (exp 2026-12-30)

We import disabled first so a wrong cert can never go live before the thumbprint gate passes — verification happens before exposure. Only then do we enable it. The subtle part: a newer latest-enabled version is not enough to serve it now — the App Gateway caches its copy and only re-fetches when its configured keyVaultSecretId changes (or on its slow 4-hour poll). So we change the link to the versioned URI to force an immediate re-pull, then change it back to versionless so future renewals auto-rotate and the config still matches terraform. Takeaway: propagation is triggered by a config change, not by the import.

The dangerous shortcut, by mechanism

The most expensive mistake is invisible on the day you make it.

"I imported the new version, now refresh the gateway."
  ├─ run `az network application-gateway update` with no changes
  │     → exit 0, looks done
  │     → BUT a no-delta update does NOT change keyVaultSecretId
  │     → AGW does NOT re-pull → keeps serving the OLD cert
  │     → silent until 2026-07-01, then ALL four hosts TLS-expired → outage
  │     ✗ WRONG  (Microsoft docs, Resolution E)
  └─ change keyVaultSecretId (versioned), then restore versionless
        → AGW re-pulls immediately → serves NEW
        ✓ CORRECT

Why it fools people: the command succeeds, so the feedback says "done." The real success signal is not exit code — it is a TLS handshake presenting the new expiry. That's why the spec makes a handshake witness mandatory and treats az exit 0 as meaningless alone.

Two more shortcuts to reject the same way:

  • Pinning to a versioned URI and leaving it: serves the right cert today but turns off autorotation — the next renewal silently won't propagate, and it drifts from terraform (which stores the versionless URI). Use versioned only as a momentary force, then restore versionless.
  • Rolling back after the old cert expires: rollback points back at the old version, which is only useful while it's still valid. After 2026-07-01 the "safe" fallback is itself expired → a late rollback is a second outage. Hence margin in days, not hours.

Why only production (the part to defend)

The challenge: "It's a wildcard — surely it's used everywhere?" The answer is label-counting backed by live reads.

Held cert SAN:  *.vpp.eneco.com   +   vpp.eneco.com
                 │                      │
   matches  X.vpp.eneco.com (ONE label)  and the apex
                 │
   does NOT match  X.dev-mc.vpp.eneco.com   (two labels)
   does NOT match  X.acc.vpp.eneco.com      (two labels)

Live Key Vault contents (read 2026-06-24):
   prod  vpp-appsec-p : wildcard-vpp-eneco-com  = *.vpp.eneco.com        ← MATCH (exp Jul 1)
   acc   vpp-appsec-a : wildcard-...            = *.acc.vpp.eneco.com    (different cert)
   acc   vpp-appsec-a : vpp-eneco-com           = acc.vpp.eneco.com      (single host)
   dev   vpp-appsec-d : wildcard-dev-mc-...     = *.dev-mc.vpp.eneco.com (different cert)
   dev   vpp-appsec-d : d-vpp-eneco-com         = dev-mc.vpp.eneco.com   (single host)

acc and dev terminate TLS with their own certificates — env-specific wildcards plus single-host certs. None is a *.vpp.eneco.com cert, and a *.vpp.eneco.com wildcard can't serve their two-label sub-domains anyway. So the renewal belongs to production and nowhere else. Honest boundary: proven across the production subscription (where vpp-ag-p is the only consumer); not swept across Sandbox/iactest — see Challenge.

Challenge — defend it

  • "How do you know it's the renewal, not coincidence?" Same CN, same SAN, same CA; the live prod object expires exactly when a renewal is due (Jul 1) and the new one starts mid-June — a textbook overlap.
  • "What would prove the scope wrong?" One consumer of a *.vpp.eneco.com cert outside vpp-ag-p — another subscription, or a non-AGW consumer (ingress/Front Door/APIM). The settling probe is a cross-subscription Resource-Graph sweep as a multi-sub identity.
  • "Where does the abstraction leak?" Versionless autorotation has a ~4h lag and the empty-update no-op quirk — exactly why we force with the versioned toggle.

Durable principle

A certificate rotation is not "done" when the import succeeds; it is done when something downstream presents the new cert. Reference the object, not the version; force the re-pull deliberately; verify by handshake, not exit code; and never let your rollback expire before your change window closes.
