The whole system in one picture

Think of one printed book (the Docker image). Three library branches each keep an order card saying which edition to stock — but all three cards live in one shared catalog (the VPP-Configuration repo, branch main). A librarian at each branch (ArgoCD) only restocks when its card changes and someone says "restock". Dev & Sandbox librarians restock automatically; ACC and PROD restock only by hand. The publisher's clerk (the CI pipeline) rewrites the order cards after each print run — and right now the clerk's instructions say rewrite all three cards at once.

Q1 — Separate configuration paths and deployment targets?

"ACC and PROD are using separate configuration paths and deployment targets."

YES for config & downstream cluster isolation = invariant to confirm

Config files and downstream Event Hubs are genuinely separate. Whether they run on separate clusters is not provable from the repo — it relies on ACC and PROD being distinct ArgoCD instances (note under the table).

Simply: ACC and PROD never share a config file and don't talk to the same Event Hub. The one thing the repo can't show you is that they run on different clusters — that has to be true outside git.

Q2 — Can ACC changes automatically propagate to PROD?

"Changes deployed to ACC cannot automatically propagate to PROD."

NEEDS ATTENTION — true for deploy, NOT true for git state

Nothing auto-runs in prod — but an ACC build does auto-rewrite the prod desired-state in git. That gap is exactly what your team observed.

Two different questions hide inside this one. "Does prod auto-deploy?" → No. "Does an acc-intended build auto-change what prod will deploy next?" → Right now, yes.

What is safe

  • ACC and PROD ArgoCD are manual-sync — neither acc nor prod has an automated: syncPolicy (only dev-mc:23 and sandbox:31 do). So a git change does not roll out on its own; a human must click Sync on the prod cluster. [A1]
  • The pipeline's only direct-to-cluster HelmDeploy is gated to the develop branch → the DEV cluster vpp-aks01-d. ACC/PROD never receive a direct push. [A1]

What is NOT safe (the real finding)

The CI step "Update values-override.yaml" has no branch condition and loops over dev/acc/prod, writing the new image tag to every folder whose flag is true, then commits to VPP-Configuration/main. (The pipeline is trigger: none, so builds are started manually or by another pipeline — not on every commit — but every build that runs does this.) The flags live in one shared variable group ape-prediction (id 604), and today they are:

test-env  = "true"
acc-env   = "true"
prod-env  = "true"     # ← every build also stamps the PROD folder
# (variable group last modified 2026-06-25 — the day before this request)

So a build meant for ACC writes the same tag into prod/values-override.yaml in the same commit. PROD won't run it until someone syncs — but the prod desired-state is already changed, and the prod app shows OutOfSync with the ACC-intended tag. Any routine prod sync then ships it. [mechanism A2 from pipeline A1 + live flags A1]

Source-of-truth note: ArgoCD reconciles from the git values-override.yaml file, not from the variable group. The group's apeprediction value (last built tag, currently 1.193) can run ahead of the git desired-state (≤1.176) — the two are written by separate pipeline steps and are not in lockstep. Read the git override file to know what prod will actually deploy. [A1]

So what should you do before the breaking changes?

Your structural separation is sound. The thing to decide is how an ACC build is allowed to touch the PROD desired-state.

Don't just flip prod-env=false. The flags are shared singleton state on group 604, read by every run — toggling them per build is a race (two builds in flight, or a forgotten reset, silently mis-target), and leaving prod-env=false would freeze prod promotion (the prod override would never advance again). The all-true setting is most likely the intended model: one build feeds all envs in git, dev auto-syncs, acc/prod wait for a human.

Pick one of these instead:

  1. Make env selection config-as-code or per-run — not shared mutable state. Split into per-env pipelines/stages each with its own variable group, OR make the env a queue-time pipeline parameter, so an ACC build cannot write the PROD folder by construction.
  2. Or keep the model and make the manual prod sync the deliberate, audited gate. ACC/PROD are already manual-sync. Before each prod sync, review the prod/values-override.yaml diff and the prod OutOfSync view; sync prod only when the tag is the one you intend. Document that the git coupling is intentional and the manual sync IS the control.
  3. Confirm the out-of-repo invariants & live state. Verify ACC and PROD are truly separate ArgoCD instances/clusters (the repo can't prove it; identical app name+namespace would collide if co-located). Re-read from the live pipeline + group — local checkouts here are Nov-2025 pipeline / Apr-2026 config. [A3]

Check yourself

  1. If a developer merges to ACC and never touches PROD, can PROD's running version change?
    answerNot on its own — prod is manual-sync. But the build will have already written the new tag into prod/values-override.yaml (prod-env=true), so the next prod sync would ship it. The exposure is in git, not in an auto-deploy.
  2. Where is ACC↔PROD isolation strongest, and where is it weakest?
    answerStrongest at runtime: separate Event Hubs and manual prod sync (and separate ArgoCD instances/clusters — to confirm). Weakest at CI: one shared variable group writes all flagged env folders together.
  3. What single change most reduces the "ACC appeared in PROD" risk?
    answerMake env selection structural — per-env pipelines/groups or a queue-time parameter — so an ACC build can't write the PROD folder. (Don't just toggle prod-env on the shared group: it races and freezes prod promotion.)
